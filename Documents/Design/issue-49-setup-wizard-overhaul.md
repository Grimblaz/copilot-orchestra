# Design: Multi-Phase Setup Wizard Overhaul + Reusable Scaffolding

**Issue**: #49
**Date**: 2026-03-03
**Status**: Finalized

## Summary

Rewrite `setup.prompt.md` from a 2-stage instruction list into a 6-phase wizard that generates
repo scaffolding files directly, extracts reusable file-operation rules into a shared instruction
file, and adds skip gates at each phase so re-runs are safe.

## Context

The original setup prompt (`setup.prompt.md`, 81 lines) gave static instructions for two stages:
copy agents into a repo and configure VS Code settings. It produced no files and had no guard
against re-running on an already-configured repo. Patterns from a separate project (Countdown-Clash)
introduced useful file-operation and issue-creation rules that should live in the template.

## Design Decisions

### 1. Setup Approach — Wizard Generates Scaffolding

**Decision**: The wizard generates repo files (.gitignore, .vscode/\*, Documents/ structure)
instead of only printing instructions.

**Rationale**: Users get a working repo state after a single prompt run; no manual copy-paste.
Ask-before-overwrite guards make re-runs safe.

### 2. Phase Structure

**Decision**: 6 phases with skip gates at Phases 1, 2, 3, 4 (conditional), and 5.

| Phase | Name | Focus | Skip Gate |
|-------|------|-------|-----------|
| 0 | Prerequisites Check | Detect code, pwsh, git, gh CLI versions | None — always runs |
| 1 | User Setup | Set `WORKFLOW_TEMPLATE_ROOT` env var (machine-level) | Already set to a valid path |
| 2 | Project Basics | Generate `.github/copilot-instructions.md` | File already exists (keep or regenerate) |
| 3 | Architecture & Conventions | Generate `.github/architecture-rules.md` | File already exists (keep or regenerate) |
| 4 | Commands | Collect build / run / test / lint commands | All of Phases 2, 3, and 5 are skipped |
| 5 | Project Scaffolding | Generate `.gitignore`, `.vscode/`, `Documents/` files | User opts out of scaffolding |

> **Phase 5 sub-steps**: 5a `.gitignore` additions · 5b `.vscode/settings.json` · 5c `.vscode/extensions.json` · 5d browser tools config + `browser-tools.instructions.md` (web only) · 5e `Documents/` structure
>
> **Note (issue #55 update)**: 5d now configures VS Code 1.110+ native browser tools as primary (via `workbench.browser.enableChatTools` setting) and offers Playwright MCP as an optional fallback. `browser-mcp.instructions.md` renamed to `browser-tools.instructions.md`.

Phase 0 uses `code --version`, `pwsh --version`, `git --version`, and `gh --version` to detect
prerequisites automatically rather than requiring the user to check manually.

### 3. Shared Instruction File — `safe-operations.instructions.md`

**Decision**: Extract File Operation Rules and Issue Creation Rules from Countdown-Clash into
`.github/instructions/safe-operations.instructions.md` and load it globally via
`chat.instructionsFilesLocations`.

**Options considered**:

| Option | Pros | Cons |
|--------|------|------|
| Keep rules inline in each agent | No extra file | Duplication across agents |
| Shared instruction file (chosen) | Single source of truth; additive load | Requires VS Code setting |
| Embed in copilot-instructions.md | Already loaded globally | Mixes concerns, grows file |

**Loading**: Added to `chat.instructionsFilesLocations` in `.vscode/settings.json` — loaded
additively alongside individual agent `instructions:` lists.

### 4. browser-tools.instructions.md — Per-Project Generated File

**Decision**: `browser-tools.instructions.md` is generated by Phase 5d for web projects only.
It is NOT a permanent shared instruction in the template repo.

**Rationale**: Browser tool configuration is project-specific. Keeping it out of the template
prevents non-web projects from getting irrelevant instructions.

**Update (issue #55)**: File renamed from `browser-mcp.instructions.md` to `browser-tools.instructions.md`. Content now covers VS Code 1.110+ native browser tools as the primary path and Playwright MCP as an optional fallback, rather than Playwright MCP as the sole path.
